# App Harness Contract and Receipt

## Project contract

The installed `.app-harness/contract.json` identifies:

- schema and harness versions;
- project root and shared engine path;
- project and personal skill locations;
- supported text-output formats;
- source-preservation and uncertain-review rules;
- default external-effects policy.

Do not turn the contract into a second PRD. Project-specific job and acceptance details can be added under `project` without deleting the installation fields.

## Receipt minimum

The deterministic analyzer already writes a receipt inside its review packet. Treat that as
the only receipt when analysis is the whole job. Create a project-level receipt under
`.app-harness/receipts/` only for a broader build, and reference the analyzer receipt rather
than copying its claims.

```json
{
  "schema_version": "1.0.0",
  "status": "pass|fail|inconclusive",
  "job": "short concrete job",
  "input_hashes": {},
  "artifacts": {},
  "tests": [],
  "ai_output_review": "pass|fail|not_applicable|not_evaluated",
  "unresolved": [],
  "external_effects": {
    "network_calls": 0,
    "secrets_accessed": 0,
    "remote_jobs_submitted": 0,
    "external_spend_usd": 0
  }
}
```

Use content hashes for replayable identity. Timestamps may describe a run but must not be the only identity key.

## Evidence vocabulary

- `measured`: produced by an executed test or inspected artifact.
- `corroborated`: supported by a separate attributable source with a scoped relationship.
- `agent_review`: generated by Codex or Claude without deterministic execution.
- `not_evaluated`: no sufficient evidence was produced.
- `not_applicable`: the criterion does not apply to this app or change.

Generated fixtures can prove code paths and invariants. They cannot establish human preference, product demand, production performance, or transfer to an inaccessible model.
